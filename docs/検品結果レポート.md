- 破壊内容: `Item::isSold()` を「購入レコードの存在確認」から「常にfalseを返す」へ一時変更した。

  - `MylistTest::test_sold_liked_items_are_marked_as_sold`
  - `PurchaseTest::test_purchased_item_is_marked_as_sold_on_item_list`
- REDの理由:
  - ItemIndexTestとMylistTestは `isSold()` がfalseを返したため、対象の真偽値アサーションが失敗した。
  - PurchaseTestは購入後のHTMLに `Sold` が表示されず失敗した。
- 判定: すべてSold判定ロジックを正しく経由した失敗。構文エラーやDB接続エラー等の無関係な例外ではない。
- 復元: `return $this->purchases()->exists();` へ復元済み。

### 検査2: 購入レコード保存ロジック

## ER図で必ず修正する差分

### 1. `item_category` テーブル

- `id` カラムを削除する。
- 正しいカラムは `item_id` と `category_id` の2つ。
- `item_id` と `category_id` はどちらもFKかつNOT NULL。
- `item_id + category_id` の組み合わせにUNIQUE制約がある。
- `created_at` と `updated_at` はない。

### 2. `likes` テーブル

- `id` カラムを削除する。
- 正しいカラムは `user_id` と `item_id` の2つ。
- `user_id` と `item_id` はどちらもFKかつNOT NULL。
- `user_id + item_id` の組み合わせにUNIQUE制約がある。
- `created_at` と `updated_at` はない。

### 3. 画像版ER図で抜けているタイムスタンプ

次のテーブルに `created_at` と `updated_at` を追加する。

- `users`
- `items`
- `categories`
- `comments`
- `purchases`

`item_category` と `likes` には追加しない。

### 4. `users` テーブで画像版ER図から抜けているカラム

- `remember_token` varchar(100), NULL可
- `two_factor_secret` text, NULL可
- `two_factor_recovery_codes` text, NULL可
- Fortify設定で使用する場合の `two_factor_confirmed_at` timestamp, NULL可

実装との完全一致を優先するなら、上記もER図に追加する。Fortifyの内部カラムを業務ER図から除外する方針の場合でも、最低限 `remember_token`、`created_at`、`updated_at` はテーブル仕様書と合わせて記載する。

### 5. UNIQUE制約の表示

- `users.email`: UNIQUE
- `purchases.item_id`: UNIQUE（1商品に購入記録は最大1件）
- `item_category(item_id, category_id)`: 複合UNIQUE
- `likes(user_id, item_id)`: 複合UNIQUE

## ER図の正しいリレーション

- `users` 1 — 0..N `items`
- `users` 1 — 0..N `comments`
- `users` 1 — 0..N `likes`
- `users` 1 — 0..N `purchases`
- `items` 1 — 0..N `comments`
- `items` 1 — 0..N `likes`
- `items` 1 — 0..1 `purchases`
- `items` 1 — 0..N `item_category`
- `categories` 1 — 0..N `item_category`
- 結果として `items` と `categories` は `item_category` を介した多対多。

## ER図作成時の最終チェック

- `item_category` と `likes` に `id` が描かれていないこと。
- `item_category` と `likes` にタイムスタンプを追加していないこと。
- `users`、`items`、`categories`、`comments`、`purchases` にタイムスタンプがあること。
- `purchases.item_id` のUNIQUEにより、items→purchasesが1対0..1であること。
- 多対多の中間テーブル `item_category` が、itemsとcategoriesの両方へ接続されていること。
